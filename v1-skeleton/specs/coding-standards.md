# AI 知识库 · 编码规范 v0.2

## 1. 目的、范围与权威性

本规范适用于仓库内的 Python、TypeScript、测试、脚本、配置和知识数据处理代码。目标是让采集、分析、整理与分发流程可读、可测、可追溯，并确保外部数据和 AI 输出始终经过校验后再进入系统。

本文件保留编码规范原稿，其终态内容已合并到 `AGENTS.md` 的“编码规范”段。Agent 执行任务时以 `AGENTS.md` 为准；修改规则时必须同步更新两处。知识条目的业务字段和状态定义以 `AGENTS.md` 的“知识条目 JSON 格式”为准；机器可读 schema 建立后，以 schema 为字段契约的唯一权威来源。

本文中的“必须”“禁止”为合并到 `main` 的硬性要求；“建议”为默认实践，偏离时必须按“例外流程”记录。规则只有同时具备工具配置、CI 检查和分支保护时才视为已自动执行。

## 2. 术语定义

- **公开 Python API**：包级 `__all__` 导出的符号；未定义 `__all__` 时，模块中不以下划线开头的模块、类、函数和方法。测试函数、fixture 和仅供模块内部使用的下划线前缀符号不属于公开 API。
- **公开 TypeScript API**：从包入口或模块中 `export` 的类型、函数、类、组件和常量。
- **外部输入**：HTTP 响应、文件、环境变量、命令行参数、消息、数据库记录、第三方 SDK 返回值和 AI 输出。
- **魔法值**：参与业务分支、状态流转、协议映射、配置查找或在两个以上生产代码位置重复使用，却没有具名定义的字符串或数字。
- **生产代码**：会被运行、打包或部署的代码；不包括规则说明文档、测试数据和专门验证扫描器的脱敏 fixture。
- **真实外部写操作**：会改变 GitHub、Hacker News、Telegram、飞书或其他外部系统状态的请求。

## 3. 设计与通用要求

- 解析、校验、业务逻辑、持久化和分发必须分层，函数和模块保持单一、清晰的职责。
- 所有外部输入必须先转换为受信任的内部类型，再用于业务逻辑、文件路径、持久化或外部写操作。
- 公共接口、关键数据结构、外部输入和返回值必须有明确类型。跨模块数据优先使用 `dataclass`、`TypedDict`、TypeScript interface/type 或显式 schema。
- AI 输出一律视为不可信输入。禁止直接将其作为代码、Shell 命令、文件路径、模板指令或外部写操作执行。
- 同一业务概念必须只有一个权威定义。状态、来源、渠道、字段名和阈值不得在 Python、TypeScript、schema 与测试中分别维护会漂移的副本。
- 设计应支持幂等执行。重跑采集、分析、整理或分发任务不得静默产生重复记录或重复消息。

## 4. Python 规范

### 4.1 格式、导入与命名

- 使用 Black 格式化并遵循 PEP 8；两者冲突时以 Black 输出为准。
- Black 版本必须由项目依赖文件锁定，配置集中放在 `pyproject.toml`，至少指定 Python 3.12 目标版本。禁止开发者自行使用不同格式参数。
- 导入按标准库、第三方库、项目内部模块分组；禁止通配符导入。
- Python 模块、源文件、函数和变量使用 `snake_case`，类使用 `PascalCase`，常量使用 `UPPER_SNAKE_CASE`。知识条目数据文件使用整理 Agent 定义的 `{date}-{source}-{slug}.json` 命名规范。
- 禁止裸 `print()`；使用标准库 `logging` 或项目统一 logger，并使用参数化日志消息。

### 4.2 类型与接口

- 公开函数必须标注全部参数与返回值类型；关键内部函数和数据边界也必须有类型。
- 优先使用 Python 3.12 原生类型语法，例如 `list[str]`、`dict[str, int]` 和 `str | None`。
- `Any` 只能出现在无法直接建模的输入边界，并必须在校验后尽快收窄。
- 类型忽略必须限定具体错误码并在同一行说明原因，例如 `# type: ignore[code]  # 原因`；无错误码或无原因的忽略禁止进入 `main`。
- 不得使用布尔参数隐藏多种行为；当调用点含义不清或分支复杂时，使用枚举、具名配置或拆分函数。

### 4.3 文档

- 公开模块、类、函数和方法必须使用 Google 风格 docstring。
- docstring 至少说明用途；存在参数、返回值或可预期异常时，分别包含 `Args`、`Returns` 和 `Raises`。
- 文档解释契约、边界条件、副作用和设计原因，不重复代码表面行为。
- 修改公开行为时必须同步更新 docstring、示例和调用方文档。

## 5. TypeScript 规范

TypeScript 规则仅在仓库存在 TypeScript 源码或 `tsconfig.json` 时启用；没有 TypeScript 项目时，CI 不得为了满足本规范创建空工程。

- `tsconfig.json` 必须设置 `"strict": true`，不得关闭 `strictNullChecks`、`noImplicitAny` 等 strict 子选项规避问题。
- 公开函数、组件属性、外部数据结构和跨模块接口必须声明类型；公开 API 必须有 TSDoc。
- 使用 `unknown` 接收不可信输入，经类型守卫或 schema 校验后再使用；禁止无理由使用 `any`。
- 避免非空断言和不安全类型断言。确需使用时，必须由同一作用域内可验证的前置条件保证安全。
- 对可辨识联合类型执行穷尽检查，使新增状态能够在编译期暴露遗漏分支。
- formatter、ESLint、TypeScript 及插件版本必须由项目依赖文件锁定，配置集中存放，禁止仅在本机忽略规则。

## 6. 常量、配置与依赖

- 魔法值必须提取为常量、枚举、配置项或 schema 定义。普通文案、参数化日志模板、测试输入和只出现一次且不参与业务判断的协议字面量可以保留在使用点。
- 环境差异配置通过环境变量或批准的配置系统注入，不得硬编码密钥、账号、内部凭据或机器专属绝对路径。
- 配置必须在程序启动或任务入口处完成类型、范围和必填项校验；缺失或无效配置应快速失败。
- 新增第三方依赖前必须说明必要性，优先使用标准库或已有依赖，并在统一依赖文件中声明版本约束。
- 运行时依赖和开发/测试依赖必须区分；CI 必须使用锁文件或可复现的受约束版本安装。

## 7. 网络、重试、异常与日志

### 7.1 网络请求

- 每个网络请求必须设置显式超时。客户端支持时分别配置连接和读取超时；否则配置总超时。
- 默认只重试连接失败、读取超时、HTTP 429 和 HTTP 5xx。HTTP 4xx 除 408、409（仅幂等冲突场景）和 429 外不得自动重试。
- 解析失败或 schema 不匹配默认不得重试；只有来源已知会短暂返回不完整响应，且有测试和注释说明时才允许有限重试。
- 重试必须同时具备最大尝试次数和最大总等待时间，使用指数退避与随机抖动，并在服务端提供有效 `Retry-After` 时遵循该值但不得突破总等待上限。
- GET、HEAD 等幂等读取可以重试。POST、PATCH、发送消息等写操作只有在使用服务端支持的幂等键或能够确认前次未生效时才可重试。
- 禁止绕过访问控制、robots 规则、服务条款或速率限制。

### 7.2 异常与日志

- 不可恢复错误应尽早失败并通过异常链保留原因；禁止空 `except`、吞掉异常或返回含义不明的默认成功值。
- 自定义异常应表达业务失败类型；调用方只捕获能够处理的具体异常。
- 日志必须包含可定位问题的对象标识和处理阶段，但不得记录 API Key、Token、Cookie、Webhook、私钥、Authorization header、完整敏感响应或个人敏感信息。
- 记录第三方异常前必须考虑其消息、URL和响应体是否含有凭据；必要时记录脱敏后的自定义字段，而不是直接输出完整异常对象。

## 8. 数据、schema 与原始文件

- 机器可读知识条目 schema 应存放在 `schemas/knowledge_article.schema.json`，使用 JSON Schema，并通过明确的 `schema_version` 字段进行版本化。
- schema 建立前，`AGENTS.md` 中的字段契约是临时权威来源；建立后，字段枚举、必填性和类型只在 schema 中定义，文档通过示例解释而不重复维护完整枚举。
- schema 的破坏性变更必须提高主版本、提供可审计迁移脚本和回滚方案；兼容性新增字段提高次版本。
- `knowledge/articles/` 只能写入通过当前目标 schema 校验的 UTF-8 `.json` 文件。
- `knowledge/raw/` 原则上只追加。写入必须先在同目录创建临时文件、刷新并原子重命名，避免进程中断产生半文件。
- 原始记录必须保存规范化来源 URL、采集时间和内容哈希。路径冲突时比较稳定 ID 与内容哈希：内容相同视为幂等成功，内容不同则写入带版本标识的新文件并记录冲突，禁止静默覆盖。
- 稳定 ID 必须由规范化后的稳定输入生成，不得使用运行时随机值。规范化算法变化时必须提供兼容或迁移策略。
- 摘要、标签、评分和分析结论必须能追溯到原始文件、来源 URL、分析模型和分析时间；无法核实的值使用 `null` 或明确标记为推断。

## 9. 分发与 dry-run

- 本地开发和测试默认启用 dry-run。dry-run 可以读取经过批准的输入和生成待发送 payload，但不得调用外部写接口、上传附件、更新远端状态或把本地条目推进为 `published`。
- dry-run 输出同样必须脱敏，不得泄露收件人隐私、凭据或不应进入日志的完整消息内容。
- 每个目标渠道必须单独记录 `pending`、`published` 或 `failed` 状态、幂等键、外部消息 ID（如有）、尝试次数、最后尝试时间和脱敏错误摘要。
- 只有所有选定渠道均成功时，条目总状态才可推进为 `published`。部分渠道成功时保留可恢复状态，只重试失败渠道，不得重复发送成功渠道。
- 真实分发必须具有明确授权；授权应限定环境、渠道和任务范围，不得由模型输出自行扩大权限。

## 10. 测试规范

- 单元测试总体覆盖率必须不低于 80%，并启用分支统计；新增或变更代码行覆盖率必须不低于 90%。覆盖范围只统计生产代码，排除测试、生成代码、缓存和第三方代码。
- 状态流转、schema 校验、稳定 ID、原子写入、重试边界、幂等分发和权限检查必须覆盖关键分支；覆盖率数字不能替代有效断言。
- Bug 修复必须先补充或同时补充能够复现问题的回归测试。
- 测试必须可重复、相互隔离，不依赖执行顺序、真实时间、随机数、网络结果或开发者本机状态；时间和随机性必须通过注入、mock 或固定 seed 控制。
- 单元测试默认禁止所有真实公网访问，包括只读请求。外部调用必须使用 mock、fixture 或录制的脱敏响应。
- 集成测试必须使用显式标记，并且只连接批准的沙箱。CI 默认任务不得持有生产写凭据。
- 测试数据必须最小化，不包含真实凭据或敏感信息；测试名称应描述场景和预期结果。

## 11. 禁止项

- 禁止将 `TODO`、`FIXME`、`HACK`、`XXX` 或等价的未完成标记提交到 `main`。后续工作必须创建可追踪任务，当前变更保持完整可用。
- 禁止注释掉的死代码、临时调试代码、生成缓存、日志文件或本机环境文件进入 `main`。
- 禁止裸 `print()`、空 `except`、无限循环、无限重试和无超时网络调用。
- 禁止在代码、测试、fixture、文档、日志或提交记录中存放真实凭据。
- 禁止把未经 schema 校验的数据写入 `knowledge/articles/`，或把失败、拒绝、部分分发成功及未经审核的条目标记为 `published`。
- 禁止测试或模型输出触发未经明确授权的外部写操作。

禁止项扫描只针对项目维护的生产代码和配置；规则说明文档、第三方代码、生成代码以及专门测试扫描器的脱敏 fixture 可以通过集中配置排除。排除列表必须接受代码评审，禁止在单个文件中临时绕过。

## 12. 工具链与 CI 验收合同

### 12.1 统一工具

- Python formatter：Black。
- Python linter：Ruff。
- Python 类型检查：mypy。
- Python 测试与覆盖率：pytest、pytest-cov、coverage.py。
- 变更代码覆盖率：diff-cover。
- TypeScript formatter：Prettier；linter：ESLint；类型检查：TypeScript compiler。
- 凭据扫描：仓库统一配置的 secret scanner；引入 CI 时必须明确产品、版本和基线文件，禁止只依赖正则表达式。

所有工具版本必须锁定在项目统一依赖文件中。工具或规则升级必须作为独立、可评审的变更，避免与业务代码修改混合。

### 12.2 标准检查

项目引入对应工具配置后，CI 至少执行以下等价检查：

```bash
python -m black --check .
python -m ruff check .
python -m mypy .
python -m pytest --cov --cov-branch --cov-report=xml --cov-report=term-missing --cov-fail-under=80
diff-cover coverage.xml --fail-under=90
```

存在 TypeScript 项目时还必须执行项目锁文件对应的 formatter check、ESLint 和：

```bash
npx tsc --noEmit
```

CI 还必须执行 schema/data contract 测试、禁止项扫描和凭据扫描。命令应由单一任务入口封装，例如 `make check` 或仓库脚本；本地与 CI 调用同一入口，禁止维护两套逻辑。

### 12.3 合并门槛

合并到 `main` 前必须满足：

- 格式、lint、静态类型、schema、数据契约、禁止项和凭据扫描全部通过。
- 所有自动化测试通过，启用分支统计后的全局覆盖率不少于 80%，变更代码行覆盖率不少于 90%。
- 新增公开接口具有类型、文档和测试；公共行为、依赖、配置或 schema 变化已同步更新文档。
- 外部输入已校验；网络调用具有超时和合规的有界重试；外部写操作具有权限与幂等保护。
- `main` 启用分支保护，禁止直接推送，并要求上述 CI checks 和至少一名 reviewer 通过。管理员绕过必须留有审计记录并按例外流程处理。

## 13. 例外与变更

- 临时例外必须记录在 PR 描述或仓库约定的例外文件中，包含规则、最小适用范围、原因、风险、负责人、关联任务和明确到日期的移除期限。
- reviewer 必须显式批准例外；CI 可识别的规则应使用集中配置和精确范围排除，不得使用散落的关闭注释。
- 过期例外视为 CI 失败，负责人必须移除例外或重新评审，不得自动延期。
- 凭据安全、数据可追溯性、未经授权的外部写操作和 `knowledge/raw/` 禁止静默覆盖等红线不接受例外。
- 修改本规范时，必须同步更新 `AGENTS.md` 的“编码规范”，并评估 CI、工具配置、schema、模板和开发文档是否需要更新。

## 14. 落地顺序

在当前仓库尚未配置完整工具链时，按以下顺序落地，避免文档先于能力而产生虚假保证：

1. 建立统一 Python 依赖和 `pyproject.toml`，锁定 Black、Ruff、mypy、pytest、coverage 与 diff-cover。
2. 建立机器可读知识条目 schema 及 schema/data contract 测试。
3. 建立单一 `check` 入口和 CI，启用格式、lint、类型、测试、覆盖率、禁止项与凭据扫描。
4. 为 `main` 启用分支保护和 required checks。
5. 接入 TypeScript 时再添加对应配置和 CI，不创建空 TypeScript 工程。
6. 工具链全部通过后，将本规范的硬性要求作为不可绕过的合并门槛。
